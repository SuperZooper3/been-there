# Capacitor A2 Implementation Plan

Decision rationale and the Capacitor vs React Native comparison are in [`docs/native-app-decision.md`](docs/native-app-decision.md).

---

## Architecture overview

Option A2: the Capacitor native shell loads the live Vercel deployment (`https://been-there.vercel.app`) in a `WKWebView` / `WebView`. All server-side routes, SSR, cookie auth, and RLS enforcement remain exactly as they are today. The Capacitor JS bridge is injected by the native runtime into whatever page is loaded, so native plugin calls work even though the JS is served from Vercel.

```
Web browsers  ──────────────────────────────────────────  Vercel (unchanged)
                                                               │
iOS/Android Capacitor shell  ──  WKWebView/WebView  ──  loads Vercel URL
                                       │
                          Capacitor JS bridge (injected)
                                       │
               @capacitor-community/background-geolocation (native)
                     iOS: CLLocationManager background mode
                     Android: FusedLocationProviderClient foreground service
```

**Why API routes are not a problem**: `createServerClient()` in the API routes uses the anon key + server-side cookie session. RLS is enforced by Supabase against the authenticated user. The WebView sends cookies on every request to the Vercel origin exactly like a browser does. Nothing changes.

**Why SSR is not a problem**: The Vercel deployment continues running Next.js App Router SSR. The native app benefits from it automatically.

## Phase 2 — Background geolocation plugin

### Install

```bash
npm install @capacitor-community/background-geolocation
npx cap sync
```

MIT licensed, free, 27.9K weekly downloads, last updated Aug 2025. GitHub: `capacitor-community/background-geolocation`.

### Changes to `components/MapApp.tsx`

The current implementation uses a polling loop with repeated `getCurrentPosition` every ~60 seconds. Replace the `startTracking` / `stopTracking` / tracking `useEffect` block with platform-aware logic.

**Add at the top of the file:**

```typescript
import { Capacitor } from '@capacitor/core';
import BackgroundGeolocation from '@capacitor-community/background-geolocation';
```

**Replace `startTracking()`** — current version calls `getCurrentPosition` once, sets `isTracking`, and relies on the interval effect. New version branches on platform:

```typescript
async function startTracking() {
  if (Capacitor.isNativePlatform()) {
    BackgroundGeolocation.addWatcher(
      {
        backgroundMessage: 'Recording your path',
        backgroundTitle: 'Been There',
        requestPermissions: true,
        stale: false,
        distanceFilter: 15, // metres — avoids noise when stationary
      },
      (location, error) => {
        if (error || !location) return;
        applyLocation(location.latitude, location.longitude);
      }
    ).then((watcherId) => {
      nativeWatcherIdRef.current = watcherId;
      setIsTracking(true);
      setTrackingDenied(false);
    }).catch(() => setTrackingDenied(true));
  } else {
    // existing web path — unchanged
    if (!navigator.geolocation) { setTrackingDenied(true); return; }
    navigator.geolocation.getCurrentPosition(
      (pos) => {
        applyLocation(pos.coords.latitude, pos.coords.longitude);
        setIsTracking(true);
        setTrackingDenied(false);
        trackingElapsedRef.current = 0;
        setTrackingProgress(0);
      },
      () => setTrackingDenied(true),
      { enableHighAccuracy: true, timeout: 10_000 }
    );
  }
}
```

**Replace `stopTracking()`**:

```typescript
function stopTracking() {
  if (Capacitor.isNativePlatform() && nativeWatcherIdRef.current) {
    BackgroundGeolocation.removeWatcher({ id: nativeWatcherIdRef.current });
    nativeWatcherIdRef.current = null;
  }
  // existing cleanup (clear interval, reset progress) stays
  setIsTracking(false);
  ...
}
```

**Add `nativeWatcherIdRef`** alongside the existing refs:

```typescript
const nativeWatcherIdRef = useRef<string | null>(null);
```

**The existing 60-second polling `useEffect`** should be conditioned on `!Capacitor.isNativePlatform()` so it only runs on web:

```typescript
useEffect(() => {
  if (!isTracking || Capacitor.isNativePlatform()) return;
  // ... existing interval logic unchanged ...
}, [isTracking]);
```

**`applyLocation()` is unchanged** — it already does H3 snapping + line interpolation + `handleCellPaint`. The native plugin just calls it more frequently and from the background.

## Phase 3 — Offline GPS buffer

The goal: GPS cells collected while offline are stored locally and flushed to the server when connectivity returns. The visual state (the `visitedCells` Set in React) already updates immediately — the user sees cells painted regardless. Only the server sync is deferred.

### How the current batch write works

`MapApp.tsx` accumulates painted cells in `pendingPaintRef` (a `Set<string>`) and a `useEffect` with a 500ms debounce posts the batch to `POST /api/cells`. Erased cells similarly go to `DELETE /api/cells`.

### Offline buffer approach

Add a localStorage-backed pending queue that the batch writer falls back to when offline. On app load and on `online` events, flush the queue.

**New helper (can live inline in `MapApp.tsx` or in `lib/offline-buffer.ts`):**

```typescript
const OFFLINE_KEY = 'bt_offline_cells';

function getOfflineQueue(): string[] {
  try { return JSON.parse(localStorage.getItem(OFFLINE_KEY) ?? '[]'); }
  catch { return []; }
}

function appendOfflineQueue(cells: string[]) {
  const existing = new Set(getOfflineQueue());
  cells.forEach(c => existing.add(c));
  localStorage.setItem(OFFLINE_KEY, JSON.stringify([...existing]));
}

function clearOfflineQueue() {
  localStorage.removeItem(OFFLINE_KEY);
}
```

**Modify the batch flush in `MapApp.tsx`:**

```typescript
async function flushPendingCells(cells: string[]) {
  if (!navigator.onLine) {
    appendOfflineQueue(cells);
    return;
  }
  try {
    await fetch('/api/cells', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ cells }),
    });
  } catch {
    appendOfflineQueue(cells);
  }
}
```

**Flush on reconnect (in a `useEffect` at component mount):**

```typescript
useEffect(() => {
  async function syncOfflineQueue() {
    const pending = getOfflineQueue();
    if (pending.length === 0) return;
    try {
      await fetch('/api/cells', {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ cells: pending }),
      });
      clearOfflineQueue();
    } catch { /* leave queue, try again next time */ }
  }

  syncOfflineQueue(); // attempt on load
  window.addEventListener('online', syncOfflineQueue);
  return () => window.removeEventListener('online', syncOfflineQueue);
}, []);
```

**No conflict resolution needed**: cells are upserted with `onConflict: 'user_id,h3_index'` and `ignoreDuplicates: false`. Re-sending a cell that's already on the server just refreshes `last_visited_at`. AND-ing everything together is already the data model.

**Photos**: offline photo uploads are not buffered in this phase. If the user tries to pin a photo offline, the existing error handling in `GeoUploadDialog` surfaces the failure. Full offline photo support is a future enhancement.

### iOS (via friend's Apple Developer account)

Two distribution options without App Store review:

**Option 1 — TestFlight (recommended, up to 10,000 testers)**
1. Friend adds your Apple ID as an App Store Connect user with Developer role
2. In Xcode: set bundle ID `com.beenthere.app`, select the team
3. Archive: Product > Archive
4. Upload to App Store Connect via Xcode Organizer
5. Add testers in App Store Connect > TestFlight
6. Apple does a brief TestFlight compliance review (~1-2 days, lighter than full review)

**Option 2 — Ad-hoc distribution (up to 100 registered devices, no review)**
1. Register target device UDIDs in the Apple Developer portal
2. Create an Ad Hoc provisioning profile including those UDIDs
3. Archive and export with the Ad Hoc profile
4. Distribute the `.ipa` via AirDrop, Apple Configurator, or a simple web download link

For a personal project with a small group, Ad-hoc is simpler (no review at all). TestFlight is better if you want to distribute to more people over time.

## Web and browser compatibility guarantees

This section documents every place the existing web implementation must be preserved exactly, and the exact patterns to use so nothing regresses.

### 1. Use a dynamic import for the background geolocation plugin

`MapApp.tsx` is `"use client"`, so no SSR risk there. But `@capacitor-community/background-geolocation` registers native plugin bindings at import time. If it is imported at the top of the file, its registration code runs in every browser that loads the page and may produce console errors or unexpected behaviour on web.

The safe pattern is a dynamic import, only executed on native:

```typescript
// Top of MapApp.tsx — safe to import statically; returns false in any browser
import { Capacitor } from '@capacitor/core';

// Never imported at the top level. Only loaded when Capacitor.isNativePlatform() is true.
async function startNativeTracking(applyLocation: (lat: number, lng: number) => void) {
  const { BackgroundGeolocation } = await import('@capacitor-community/background-geolocation');
  const id = await BackgroundGeolocation.addWatcher(
    { backgroundMessage: 'Recording your path', backgroundTitle: 'Been There',
      requestPermissions: true, stale: false, distanceFilter: 15 },
    (location, error) => {
      if (error || !location) return;
      applyLocation(location.latitude, location.longitude);
    }
  );
  return id; // store in nativeWatcherIdRef
}
```

`@capacitor/core` itself is safe to import statically — it is designed to be a no-op on web and returns sensible defaults (`Capacitor.isNativePlatform() === false`, `Capacitor.getPlatform() === 'web'`).

### 2. Guard the existing 60-second polling `useEffect` against running on native

The current effect at lines 355–376 of `MapApp.tsx` fires whenever `isTracking` is true. On native, the plugin already fires `applyLocation` via its own listener. If this effect also runs, you get double-painting and double battery drain. One line at the top of the effect prevents this:

```typescript
useEffect(() => {
  if (!isTracking || Capacitor.isNativePlatform()) return; // ← add this guard
  const INTERVAL_MS = 60_000;
  // ... rest of the existing effect completely unchanged ...
}, [isTracking]);
```

The web path inside this effect — `navigator.geolocation.getCurrentPosition` calls, the progress ticker, the `trackingElapsedRef` countdown — stays byte-for-byte identical. Nothing is removed.

### 3. `applyLocation()` is the single convergence point — do not touch it

```typescript
// lines 305–319 — unchanged in both web and native paths
function applyLocation(lat: number, lng: number) {
  setCurrentLocation({ lat, lng });
  const newCell = snapToCell(lat, lng);
  if (prevLocationRef.current) {
    const prevCell = snapToCell(prevLocationRef.current.lat, prevLocationRef.current.lng);
    const pathCells = getCellsAlongLine(prevCell, newCell);
    pathCells.forEach((cell) => handleCellPaint(cell));
  } else {
    handleCellPaint(newCell);
  }
  prevLocationRef.current = { lat, lng };
}
```

Both the web `getCurrentPosition` callback and the native plugin callback call `applyLocation(location.latitude, location.longitude)` with the same two-number signature. `prevLocationRef`, cell interpolation, and `handleCellPaint` are all shared and untouched.

### 4. `startTracking()` and `stopTracking()` stay callable from the same UI

`handleTrackToggle()` (line 336) calls `startTracking()` or `stopTracking()` — this wiring does not change. The platform branch is internal to those two functions. From the toolbar and any other call site, the API is identical.

```typescript
// handleTrackToggle — unchanged
function handleTrackToggle() {
  if (isTracking) { stopTracking(); }
  else if (trackingDenied) { setShowDrawModal(true); }
  else { startTracking(); }
}
```

### 5. `trackingProgress` UI — hide on native, keep on web

The 60-second progress ring makes sense on web (one GPS sample per minute). On native, the plugin fires on movement (no fixed interval), so the progress bar is meaningless. Wherever `trackingProgress` is rendered in the toolbar, add:

```typescript
{!Capacitor.isNativePlatform() && trackingProgress > 0 && (
  <TrackingProgressRing value={trackingProgress} />
)}
```

The `trackingProgress` state and all its web-side logic stay intact — it just is not rendered on native. No existing web UI is removed.

### 6. `stopTracking()` — additive only

The existing `stopTracking()` body:

```typescript
function stopTracking() {
  setIsTracking(false);
  setCurrentLocation(null);
  setTrackingProgress(0);
  trackingElapsedRef.current = 0;
  prevLocationRef.current = null;
  if (trackingTimerRef.current) { clearInterval(trackingTimerRef.current); trackingTimerRef.current = null; }
}
```

On native we add one block before the existing lines — the plugin watcher removal. Everything else stays:

```typescript
function stopTracking() {
  // native-only addition
  if (Capacitor.isNativePlatform() && nativeWatcherIdRef.current) {
    import('@capacitor-community/background-geolocation').then(({ BackgroundGeolocation }) => {
      BackgroundGeolocation.removeWatcher({ id: nativeWatcherIdRef.current! });
    });
    nativeWatcherIdRef.current = null;
  }
  // existing lines — unchanged
  setIsTracking(false);
  setCurrentLocation(null);
  setTrackingProgress(0);
  trackingElapsedRef.current = 0;
  prevLocationRef.current = null;
  if (trackingTimerRef.current) { clearInterval(trackingTimerRef.current); trackingTimerRef.current = null; }
}
```

### 7. `trackingDenied` and the draw modal — unchanged on both platforms

`setTrackingDenied(true)` is called when permission is denied. On native, the plugin's `requestPermissions: true` option handles the OS permission prompt. If the user denies it, the watcher promise rejects and we call `setTrackingDenied(true)` — same state as the web path, same modal behaviour.

### 8. `navigator.geolocation` check in `startTracking()` — web-only guard stays

The existing `if (!navigator.geolocation) { setTrackingDenied(true); return; }` line should be moved inside the web branch of `startTracking()` (since on native we don't check `navigator.geolocation` at all). Keeping it at the top of the function would short-circuit native tracking if somehow `navigator.geolocation` was unavailable in the WebView, which could theoretically vary.

### Summary table

| Existing piece | Changed? | Notes |
|---|---|---|
| `applyLocation(lat, lng)` | No | Convergence point; both paths call it |
| `handleTrackToggle()` | No | Same call sites, same signature |
| `startTracking()` web branch | No | Unchanged inside the `else` branch |
| `stopTracking()` web cleanup | No | Native block prepended; existing lines untouched |
| 60-second polling `useEffect` | One line added | `if (Capacitor.isNativePlatform()) return;` guard |
| `trackingProgress` state/logic | No | Web logic untouched; only rendering guarded on native |
| `trackingDenied` / draw modal | No | Same state machine, same UI |
| `prevLocationRef` | No | Used by `applyLocation` from both paths |
| `pendingPaintRef` / batch flush | No (Phase 3 extends it) | Offline buffer wraps the existing flush, doesn't replace it |
| All other state and UI | No | Completely untouched |

## Auth note

Sign in (email + password) works natively in the WebView with no changes. The one edge case: the password reset flow sends an email link to `https://been-there.vercel.app/auth/callback?code=...`. On iOS, this link opens in Safari (system browser), not the WebView — so the session is set in Safari, not the app. The user then needs to open the app and sign in again with their new password. This is a minor UX quirk, not a blocker. If it becomes annoying, it can be addressed later with a custom URL scheme (`beenthere://`) or the `@capacitor/browser` in-app browser plugin.

## Pre-implementation bug review

Predicted issues categorised by severity, each with the exact fix. Address these before or during the implementation phases, not after.

---

### Showstoppers — will break the build or produce obvious failures immediately

#### S1. The `out/` directory does not exist and `npx cap sync` will fail

The plan specifies `--web-dir=out` and `webDir: 'out'` in `capacitor.config.ts`. Next.js without `output: 'export'` never creates an `out/` folder — it writes to `.next/`. Capacitor CLI requires the `webDir` to physically exist at sync time, even in A2 mode where it is not actually used for serving. Running `npx cap sync` without this will throw a hard error.

**Fix**: After `npx cap init`, create the placeholder:

```bash
mkdir -p out && echo '<!DOCTYPE html><html><body></body></html>' > out/index.html
```

Or point `webDir` to `public/` which already exists and is harmless. Either way, add `out/` to `.gitignore` since it is not the real build output.

#### S2. The `removeWatcher` call in `stopTracking` reads null from a cleared ref

Not a hard build blocker — the app will run — but it is a real battery-drain bug worth getting right during Phase 2.

The plan's `stopTracking` pattern sets `nativeWatcherIdRef.current = null` synchronously, then the dynamic import resolves and passes `nativeWatcherIdRef.current!` (now null) to `removeWatcher`. The watcher is never removed and keeps firing location callbacks into a dead context.

**Be careful when writing `stopTracking`** — always capture the ID into a local variable before clearing the ref:

```typescript
function stopTracking() {
  if (Capacitor.isNativePlatform()) {
    const watcherId = nativeWatcherIdRef.current; // capture first
    nativeWatcherIdRef.current = null;            // then clear
    if (watcherId) {
      import('@capacitor-community/background-geolocation').then(({ BackgroundGeolocation }) => {
        BackgroundGeolocation.removeWatcher({ id: watcherId }); // local var, safe
      });
    }
  }
  setIsTracking(false);
  setCurrentLocation(null);
  setTrackingProgress(0);
  trackingElapsedRef.current = 0;
  prevLocationRef.current = null;
  if (trackingTimerRef.current) { clearInterval(trackingTimerRef.current); trackingTimerRef.current = null; }
}
```

#### S3. Plugin version mismatch with Capacitor major version

Not a blocker upfront — just something to check first if you hit confusing peer dependency errors or native build failures after install. `@capacitor-community/background-geolocation` targets a specific Capacitor major. If it was built for v6 and you install v7, you'll get hard errors. If the build breaks unexpectedly, run:

```bash
npm info @capacitor-community/background-geolocation peerDependencies
```

and align all `@capacitor/*` packages to the same major version the plugin declares.

### High risk — will cause real failures under normal use conditions

#### H1. `reloadOnOnline: true` in `next.config.ts` will kill an active tracking session

The current `next.config.ts` has:

```javascript
withPWA({ reloadOnOnline: true, ... })
```

This means the `next-pwa` service worker **reloads the entire page** when the device reconnects to the network. In A2 mode, the service worker from `been-there.vercel.app` is registered inside the Capacitor WebView just as it would be in a browser. When a user goes through a tunnel (offline) and comes back online while tracking:

1. The service worker fires the reload.
2. The page reloads — React state is wiped, including `isTracking`.
3. The native background watcher is still running on the native side, firing location callbacks into the old JS context, which no longer exists.
4. The UI shows tracking as stopped. The native watcher leaks indefinitely.
5. Battery drains silently.

**Fix**: Disable `reloadOnOnline` in `next.config.ts` before shipping the native app:

```javascript
withPWA({
  reloadOnOnline: false, // ← was true; kills tracking sessions on reconnect
  ...
})
```

This is safe for the web version too — refreshing on reconnect is a jarring UX pattern that most apps have moved away from.

#### H2. iOS session cookie persistence across app restarts — manually verify on real device

Cookie persistence in `WKWebView` for remote URLs (Capacitor A2) has historically been unreliable. Capacitor 5+ uses `WKWebsiteDataStore.default()` which should persist correctly, but iOS 17 App-Bound Domains can restrict cross-site data for domains not listed in `Info.plist`.

This cannot be tested in the iOS Simulator — only a real device shows the real behaviour. See the **Manual test checklist** at the end of this plan for the exact steps. If it fails, the fix is to add `been-there.vercel.app` to App-Bound Domains:

```xml
<key>WKAppBoundDomains</key>
<array>
  <string>been-there.vercel.app</string>
</array>
```

#### H3. Development iteration requires a local IP, not `localhost`, and extra iOS config for HTTP

With `server.url: 'https://been-there.vercel.app'` in `capacitor.config.ts`, the native app always loads production. You cannot iterate on code changes locally — every change requires a Vercel deploy + Xcode archive. Over a few weeks of development this adds up to enormous friction.

The fix is a dev-mode config pointing to your local machine's LAN IP:

```typescript
const DEV_SERVER = 'http://192.168.1.X:3000'; // your actual LAN IP

const config: CapacitorConfig = {
  ...
  server: {
    url: process.env.CAPACITOR_ENV === 'production'
      ? 'https://been-there.vercel.app'
      : DEV_SERVER,
    cleartext: process.env.CAPACITOR_ENV !== 'production', // allows HTTP on Android
  },
};
```

For iOS, HTTP (non-HTTPS) localhost requires an `NSAppTransportSecurity` exception in `Info.plist`:

```xml
<key>NSAppTransportSecurity</key>
<dict>
  <key>NSAllowsLocalNetworking</key>
  <true/>
</dict>
```

Run `npx cap sync` after toggling the env. **Important**: always switch back to production before building the distribution APK/IPA.

#### H4. iOS "Always Allow" location permission — show a persistent warning, let tracking continue

iOS always starts with "When In Use" on the first permission prompt regardless of what the app requests. If the user never upgrades to "Always" in Settings, background tracking silently stops the moment they leave the app. The app should not block on this — it should let tracking start with foreground-only permission and show a persistent, non-blocking warning banner when it detects that only "When In Use" was granted.

The plugin exposes the granted permission level. After `addWatcher` resolves, check the permission state:

```typescript
// After the watcher is registered:
if (permissionStatus === 'whenInUse') {
  // set a state flag, e.g. setTrackingBackgroundLimited(true)
  // render a persistent but dismissable banner:
  // "Background tracking limited — tap to enable in Settings"
}
```

Tapping the banner should open Settings directly:

```typescript
import { NativeSettings, AndroidSettings, IOSSettings } from 'capacitor-native-settings';
NativeSettings.openIOS({ option: IOSSettings.LocationServices });
```

The key behaviour: tracking proceeds regardless. The warning stays visible until the permission is upgraded or the user dismisses it. Never block or disable tracking over this.

#### H5. MapLibre WebGL context loss when iOS backgrounds the app — auto-fix in Phase 2

When the Capacitor app is backgrounded on iOS, the OS may discard the `WKWebView`'s GPU context to reclaim memory. The symptom: map goes blank on return to foreground.

This is **handled automatically** by adding a Capacitor `App` lifecycle listener in `Map.tsx` that calls `map.resize()` on foreground return. This is a planned code change in Phase 2, not a manual step.

Add `@capacitor/app` to the Phase 1 install:

```bash
npm install @capacitor/core @capacitor/cli @capacitor/ios @capacitor/android @capacitor/app
```

In `Map.tsx`, inside the `useEffect` that initialises the MapLibre instance, add after map creation:

```typescript
import { App } from '@capacitor/app';
import { Capacitor } from '@capacitor/core';

// after map is created:
if (Capacitor.isNativePlatform()) {
  App.addListener('appStateChange', ({ isActive }) => {
    if (isActive && mapRef.current) {
      mapRef.current.resize();
      // resize() recovers from a stale viewport; if WebGL context was truly
      // destroyed, this alone won't rebuild it — in that case location.reload()
      // is the fallback. Start with resize() and only escalate if needed.
    }
  });
}
```

No cleanup needed for the listener on unmount since the map component lives for the entire app lifetime.

### Medium risk — will cause subtle or intermittent bugs

#### M1. The offline buffer silently drops erase operations

The Phase 3 offline buffer only queues **paint** cells in localStorage. The `flushPending` function also handles erases (`DELETE /api/cells`). If a user erases cells while offline:

- The erase is applied to local React state immediately (cells disappear visually).
- The `DELETE` fetch fails (offline).
- The offline buffer never captures it.
- On reconnect, the sync doesn't re-issue the DELETE.
- On next app load, `GET /api/cells` returns the un-erased cells from Supabase.
- The erased cells reappear. Data inconsistency.

For GPS tracking this is unlikely (you don't erase while actively moving), but for manual draw+erase sessions it is a real case.

**Fix**: Add a parallel erase queue:

```typescript
const OFFLINE_ERASE_KEY = 'bt_offline_erases';

// In flushPending, if offline: save toErase to localStorage.
// In syncOfflineQueue, issue the DELETE as well.
// Order matters: fire DELETE before POST to avoid re-adding just-erased cells.
```

#### M2. Race condition between the reconnect flush and the 500ms batch flush

When the device comes back online, two things can fire near-simultaneously:

1. The `window.addEventListener('online', syncOfflineQueue)` handler fires and POSTs the localStorage queue.
2. The 500ms `flushPending` timer fires and POSTs `pendingPaintRef.current`.

Both fetch to `/api/cells`. Since the DB uses `onConflict` upsert, duplicate sends are safe. But `clearOfflineQueue()` is called inside `syncOfflineQueue` after a successful fetch. If the 500ms flush also writes to the offline queue concurrently (because `navigator.onLine` flips back to true between checks), cells could be double-counted in the queue and then cleared partially.

**Fix**: Use a lock flag: `let isSyncing = false`. Set it before the flush, clear it after. If `isSyncing` is true when `syncOfflineQueue` is called, skip the sync (the flush will handle it). Alternatively, use a `Promise` queue.

#### M3. The `applyLocation` function passed to the plugin callback can become stale if its captures change

`applyLocation` is a plain function defined inside the component (not `useCallback`). The native plugin callback is registered once and holds the version of `applyLocation` from the moment `startTracking` was called. Today `applyLocation` only closes over stable values (React setters, pure functions, refs), so this is safe. But if anyone adds a non-stable capture to `applyLocation` in the future (e.g., some state value), the plugin callback will use a stale snapshot of that value forever.

**Fix**: Always route the callback through a ref so it always sees the current version:

```typescript
const applyLocationRef = useRef(applyLocation);
useEffect(() => { applyLocationRef.current = applyLocation; });
// applyLocation changes every render, so no dep array — always kept current.

// In startNativeTracking:
(location, error) => {
  if (error || !location) return;
  applyLocationRef.current(location.latitude, location.longitude);
}
```

This is the correct React pattern for passing callbacks to external systems that outlive renders.

#### M4. Stadia Maps tile loading from WKWebView — the `Referer` header

Stadia Maps domain authentication works by checking the HTTP `Referer` (or `Origin`) header on tile requests. When `maplibre-gl` fetches tiles from inside a `WKWebView` loading `https://been-there.vercel.app`, it sends tile requests with `Referer: https://been-there.vercel.app/`. This is the domain you've registered with Stadia, so it should pass.

However, some WKWebView versions suppress the `Referer` header for cross-origin requests. If tiles load as blank squares but the rest of the app works, this is the cause.

**Fix**: If tile loading fails in the WebView, switch from domain auth to an API key. Add the API key to the tile URL template in the Stadia configuration, and store it as a `NEXT_PUBLIC_` env var (it is a client-side key anyway). In the Stadia dashboard, restrict the key by domain for security.

#### M5. The `out/` placeholder must be added to `.gitignore`

The `out/` directory with its dummy `index.html` (fix for S1) must not be committed or it will create confusion. It is also not the real Next.js output. Add it to `.gitignore`.

#### M6. The plan's `capacitor.config.ts` `plugins.BackgroundGeolocation` config key may not be the correct key name

The `capacitor.config.ts` plugins block uses:

```typescript
plugins: {
  BackgroundGeolocation: { notificationTitle: '...', notificationText: '...' }
}
```

The `@capacitor-community/background-geolocation` plugin's config key in `capacitor.config.ts` must match exactly what the plugin declares in its source. Some community plugins use the plugin class name, others use a custom key. Verify against the plugin's README — if the key is wrong, the notification config is silently ignored and Android may show a generic or empty notification text in the status bar.

### Low risk — minor issues or things to watch

#### L1. Android OEM battery optimisation will silently kill background tracking for some users

Android's Doze mode exempts foreground services (which the plugin uses), but Samsung One UI, Xiaomi MIUI, Huawei EMUI, and OnePlus OxygenOS have additional aggressive kill policies that terminate foreground services even with a visible notification. There is no code fix — it is a device-level restriction. Document it: users on these devices may need to whitelist the app under "Battery" → "Unrestricted" in their device settings.

#### L2. The `npx cap init` syntax varies by Capacitor version

In Capacitor v5+, `npx cap init` is interactive and does not take `--web-dir` as a CLI flag in the same position as earlier versions. The correct approach for v6/v7 is to run `npx cap init` interactively, or pass name and appId as positional args and then edit the generated `capacitor.config.ts` file to set `webDir`. Do not assume the exact command in the plan will work verbatim — run `npx cap init --help` first to see the current signature.

#### L3. `handleCellPaint`'s undo stack merge behaviour during GPS tracking creates very large undo entries

During a GPS tracking session, every `applyLocation` call fires `handleCellPaint` for potentially dozens of cells (via `getCellsAlongLine`). The undo stack logic in `handleCellPaint` merges consecutive paint actions into the last entry if it was also a paint action. Over a long tracking session, the single undo entry accumulates thousands of cells. Undoing after tracking would undo the entire tracked session at once.

This is existing behaviour but becomes much more pronounced with the native plugin firing many more updates per session than the current 1-per-60-seconds web implementation. It is not a bug per se — just worth knowing. If you want per-minute undo granularity instead, the undo stack logic would need a time-based grouping break.

#### L4. The photo picker `<input type="file">` works in WKWebView but `accept="image/*"` may not filter on older iOS

On iOS 15 and below, `accept="image/*"` on a file input inside WKWebView sometimes shows all files, not just images. This is an OS-level limitation, not a Capacitor issue. If supporting iOS 15, use `@capacitor/camera` plugin as a fallback for photo selection on native.

#### L5. The initial load fetches `/api/cells?zoom=13` — a hardcoded zoom

In `MapApp.tsx` line 75, the initial load always fetches at zoom 13, which returns res-9 cells. This is correct and intentional. On mobile, the default zoom might differ, but it is not worth changing for this implementation — just noting that the hardcoded `13` should stay and is not a bug.

#### L6. `flushPending` has no error handling — a failed write silently loses data even today

The existing `flushPending` (lines 104–124) does not check the response status or catch network errors:

```typescript
await fetch("/api/cells", { method: "POST", ... }); // error? silent loss.
```

This pre-dates the Capacitor work and is an existing issue. The offline buffer (Phase 3) partially fixes it for `navigator.onLine === false` cases, but a fetch that fails for other reasons (server error, bad request) still silently loses data. Low priority for a personal project, but worth a try/catch and a retry or a user-visible error.
