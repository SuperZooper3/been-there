# Stack, Storage, and Auth Comparison

## Stack by phase

## Phase 1 (MVP)
- Frontend/app: Next.js (App Router) on Vercel
- Map rendering: MapLibre GL JS
- Spatial index: H3 (`h3-js`), store all visited cells at resolution 9
- Database: Postgres
- Auth: simple emailless login or Auth.js provider login
- Images: object storage (Supabase Storage, Vercel Blob, or Cloudflare R2)

## Phase 2 (Auto-fill while app open)
- Add browser Geolocation API (`watchPosition`)
- Same data model and APIs as Phase 1
- Add sampling/batching policy for battery control

## Phase 3 (Import runs/routes)
- Add GPX import pipeline
- Accept Strava/Garmin export files
- Convert track points to H3 res-9 cells and bulk upsert

## Phase 4 (Friends and sharing)
- Multi-user overlays, visibility toggles, sharing links

## Phase 5 (Optional native background tracking)
- Capacitor wrapper + background location plugin

## Neon vs Supabase (and other options)

## Option 1: Neon + Auth.js + separate object storage
- What it is: serverless Postgres (Neon) plus your own auth stack plus image storage provider.
- Pros:
  - Excellent Postgres experience and branching workflows.
  - Flexible architecture and less vendor coupling.
  - Great if you already prefer Next.js + custom backend control.
- Cons:
  - More moving parts (auth + storage separate).
  - More integration work in MVP.

## Option 2: Supabase (Postgres + Auth + Storage together)
- What it is: Postgres plus built-in auth and object storage in one platform.
- Pros:
  - Fastest MVP velocity.
  - Fewer services to configure.
  - Clean image upload flow via built-in buckets.
- Cons:
  - Slightly higher platform coupling.
  - Some teams later outgrow defaults and customize anyway.

## Option 3: Vercel Postgres + Vercel Blob + Auth.js
- Pros:
  - Tightest Vercel integration.
  - Simple deployment DX.
- Cons:
  - Similar coupling tradeoff, not as Postgres-focused as Neon workflows.

## Option 4: Firebase (Firestore/Storage/Auth)
- Pros:
  - Very fast for client-first apps.
  - Mature auth/storage.
- Cons:
  - Different data model ergonomics (not SQL-first), not ideal for H3 SQL workflows.

## Recommended choice now
Given your request for simplicity and integrated auth/storage, choose:

- **Supabase for Phase 1-3** (DB + Auth + image storage)
- Keep schema and APIs portable so migration to Neon later is possible.

This gives lowest setup burden and avoids early storage complexity.

## "Can Neon store images directly?"
Not really in the way you want for app media:
- Neon is Postgres, best for metadata and relational records.
- You can technically store binary blobs in DB, but it is usually costly and inefficient for map photo workloads.
- Best practice: store image files in object storage, keep only URLs/keys in Postgres.

So if you want one integrated platform, Supabase is a practical fit.

## Service worker simple explanation
A service worker is a background browser script used by PWAs to:
- Cache app files for faster repeat loads
- Enable some offline behavior
- Support installability behavior

For your MVP, it is useful but lightweight. It does not mean "complex backend infra."

## Capacitor deployment explained simply
Capacitor wraps your web app into a real iOS/Android project.

How it works:
1. Build web app (`next build`/static output path strategy)
2. Copy build into Capacitor shell
3. Open Xcode/Android Studio projects generated by Capacitor
4. Run on device, then optionally ship to App Store / Play Store

Important:
- You do not need to release to stores immediately.
- You can sideload/test builds privately first.
- For true always-on background GPS, native packaging is usually required.
- This is additive: your data model and server APIs remain unchanged.
